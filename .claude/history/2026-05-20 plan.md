# 2026-05-20 — Plan: Allow selecting dot-folders in subdirectories

## Context

The plugin currently lets users opt-in to indexing hidden dot-folders only at the **vault root**. The settings tab calls `HiddenFoldersIndexer.listHiddenRootFolders()` which scans `adapter.list('/')` once and surfaces matches. A user with `.archive` buried under `notes/projects/` has no way to enable it through the UI.

The indexing patch itself (`listRecursiveChild` + `reconcileFile` monkey-patches) is already depth-agnostic — `isUnderPrefix` does pure string-prefix matching on normalised paths, so `enabledPrefixes` can already contain any vault-relative path. The blocker is purely **discovery** + **UI**.

This change adds a lazy, expand-on-click tree view to the settings tab so the user can drill into any subdirectory and toggle nested dot-folders.

## User-confirmed decisions

| Decision | Choice |
|---|---|
| Scan depth | Lazy expand-on-click (only fetch immediate children when the user expands a node) |
| Traversal scope | Descend into all folders (including non-dot), so dot-folders under arbitrary parents are reachable |
| UI shape | Nested tree with chevron expand/collapse |
| Always-excluded names | `.obsidian` (`vault.configDir`), `.git`, `.trash`, `node_modules` |

## Approach

Three concrete changes, in order:

### 1. Add a depth-aware listing method on the indexer

**File:** `src/app/services/hidden-folders-indexer.ts`

Add a new public method alongside `listHiddenRootFolders`:

```typescript
async listChildFolders(parent: string): Promise<{
    dotFolders: string[]      // immediate children whose name starts with '.'
    regularFolders: string[]  // immediate children that don't start with '.'
}>
```

Behaviour:
- Normalise `parent` (empty string = vault root).
- Call `this.internalAdapter().list(parent === '' ? '/' : parent)`.
- For each entry in `listed.folders`, take the **basename** (`path.split('/').pop()`).
- Skip any basename in the always-excluded set: `[vault.configDir, '.git', '.trash', 'node_modules']`. Keep this set in a private static field for reuse.
- Partition into `dotFolders` (starts with `.`) and `regularFolders`.
- Sort both alphabetically.
- Return full vault-relative paths, not basenames — the UI needs the full path for toggle wiring.

The existing `listHiddenRootFolders` becomes a thin wrapper: `(await listChildFolders('')).dotFolders` (or stays as-is for backward compatibility — both are cheap).

### 2. Fix `pathExistsOnDisk` to handle nested paths

**File:** `src/app/services/hidden-folders-indexer.ts` (lines 122–128)

Current implementation only checks vault-root folders:
```typescript
async pathExistsOnDisk(rawPath: string): Promise<boolean> {
    const rootListing = await this.internalAdapter().list('/')
    const available = rootListing.folders.map((p) => this.normalize(p))
    return available.includes(path)
}
```

This breaks for `notes/projects/.archive` because `.archive` isn't in the root listing.

Fix: list the **parent** directory of `path` and check whether `path` (full) is among the returned folders. For `path === 'notes/projects/.archive'`, list `notes/projects` and look for `notes/projects/.archive` in the result. For root-level paths the parent is `''` → `list('/')` (preserves current behaviour).

### 3. Rebuild `renderFolderList` as a lazy tree

**File:** `src/app/settings/settings-tab.ts` (lines 53–100)

Replace the flat folder loop with a recursive tree renderer. Sketch:

```typescript
private async renderFolderList(containerEl: HTMLElement): Promise<void> {
    const tree = containerEl.createDiv({ cls: 'hfa-tree' })
    await this.renderTreeNode(tree, '', /* depth */ 0)
}

private async renderTreeNode(
    container: HTMLElement,
    parentPath: string,
    depth: number
): Promise<void> {
    const { dotFolders, regularFolders } = await this.plugin.indexer.listChildFolders(parentPath)

    // Render dot-folders as togglable rows
    for (const fullPath of dotFolders) {
        this.renderDotFolderRow(container, fullPath, depth)
    }

    // Render non-dot folders as expandable rows with a chevron
    for (const fullPath of regularFolders) {
        this.renderExpandableRow(container, fullPath, depth)
    }
}
```

UI details:
- Indent via `paddingLeft: ${depth * 1.25}rem` on each row's container. No nested DOM trees needed — flat list with indentation is simpler and easier to virtualise later.
- Each non-dot row gets a chevron button (`▶` collapsed, `▼` expanded). Clicking it:
  - First click: calls `renderTreeNode(childContainer, fullPath, depth + 1)` and stores the container reference.
  - Subsequent clicks: toggle the child container's `display`. Do not re-fetch.
- Each dot-folder row uses the existing `new Setting(rowEl).setName(basename).addToggle(...)` pattern. The toggle's `onChange` calls `plugin.updateEnabledFolders` with the **full** vault-relative path — no schema changes needed on that side.
- Show the basename in the toggle label (`new Setting(...).setName(basename)`); show the full path as small muted text below for disambiguation (`setDesc(fullPath)`) — only for non-root entries to avoid noise.
- Empty-state copy when a folder has no children of either kind.

### 4. (Minor) Update the intro copy

**File:** `src/app/settings/settings-tab.ts` (lines 26–51, in `renderIntro`)

Current text says *"Only folders at the vault root are listed."* — update to reflect the new tree behaviour. Mention that scanning is lazy and the always-excluded folders.

## What stays untouched

- `PluginSettings.enabledFolders: string[]` — already a string array; storing `'notes/projects/.archive'` works without migration.
- `loadSettings()` — already filters to strings; pre-existing root-only entries continue to work.
- `enablePath` / `disablePath` / `runEnable` / `runDisable` — already accept any vault-relative path.
- The monkey-patches on `listRecursiveChild` and `reconcileFile` — already depth-agnostic via `isAnyEnabled` / `isUnderPrefix`.
- Background task pipeline (`startEnableTask`, `runBackgroundSync`, notice updates) — unchanged.

## Settings migration

None required. The `enabledFolders` field is `string[]` and the runtime already normalises paths. Users upgrading will see their previously-enabled root folders intact; new nested selections just append new full paths.

## Verification plan

1. **Test infrastructure prep:** the existing `makeApp(folders)` helper in `hidden-folders-indexer.spec.ts` (lines 19–27) is path-agnostic — `list()` returns the same folders regardless of the path argument. That's fine for the old root-only API but won't exercise nested behaviour. Upgrade `FakeAdapter.list` to accept a `Record<string, string[]>` map of path → folders, with sensible defaults for unknown paths (empty list, or throw to exercise the try/catch in `listChildFolders` and `pathExistsOnDisk`). Existing tests should still pass after the upgrade.
2. **Unit tests** (`bun test`):
   - `listChildFolders('')` returns same dot-folder set as old `listHiddenRootFolders`.
   - `listChildFolders('subdir')` returns the right partition for a nested directory (dot vs regular).
   - Exclusions: `.obsidian`, `.git`, `.trash`, `node_modules` never appear at any depth.
   - Sort order is alphabetical.
   - `pathExistsOnDisk('notes/projects/.archive')` returns `true` when present, `false` when absent.
   - `pathExistsOnDisk` returns `false` (does not throw) when the parent directory itself doesn't exist.
3. **Type + lint:** `bun run validate` (runs `tsc`, `test`, `lint` in parallel).
4. **Manual smoke test:**
   - Run `bun run dev` with `OBSIDIAN_VAULT_LOCATION` set to the test vault.
   - Reload Obsidian (Ctrl+R) → open plugin settings.
   - Verify root-level dot-folders still appear and existing toggles still work.
   - Expand a non-dot folder containing a nested `.x` → verify it appears.
   - Toggle the nested dot-folder on → confirm a Notice appears, indexing completes, files appear in the file tree.
   - Toggle off → confirm files are removed cleanly.
   - Restart Obsidian → confirm the nested enabled folder persists and re-indexes on `onLayoutReady`.
   - Verify `.obsidian`, `.git`, `.trash` never appear at any level.

## Open / deferred

- **Search/filter box** on top of the tree (helpful for huge vaults). Skipping for v1 — add only if the tree itself proves unwieldy.
- **Persisted expand state** between settings-tab visits. Skipping for v1; tree resets to all-collapsed on each open, which is acceptable for an opt-in admin UI.
- **Performance**: with lazy expand, the only synchronous cost is the initial root listing. No virtualisation needed.
